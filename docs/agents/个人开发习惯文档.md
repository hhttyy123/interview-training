# 个人开发习惯文档

## 1. 命令执行习惯

- 对于所有测试、安装、下载、构建、启动等命令，统一由我本人手动执行。
- 在需要运行命令时，必须直接给出完整命令，不能只写简略形式，也不能省略工作目录、依赖前置条件或关键参数。
- 默认不要替我自动执行测试命令、下载命令或安装命令，除非我明确要求你代为执行。

## 2. 功能验收习惯

- 每完成一个功能后，都要同步提供对应的验收方式。
- 验收说明要尽量具体，不能只写“你可以测试一下”或“运行看看”。
- 验收内容应至少包括：前置条件、执行步骤、预期结果，以及出现异常时重点检查什么。
- 如果一个功能包含多个关键路径，应分别说明每条路径的验收方式。

## 3. 产品与架构设计习惯

- 默认不要围绕少数预置选项做硬编码方案。尤其是岗位、公司、职业路径、追问策略、评分标准这类会天然扩展的能力，必须优先设计成通用、可配置、可由现场生成数据驱动的结构。
- 如果当前实现为了落地必须保留预置模板，必须明确标记为 fallback 或 seed，不能把预置模板当成产品能力上限。
- 面向“全部职业/全部岗位可用”的功能，应优先抽象为能力原型、职业族、证据槽、策略库、动态模型，而不是写死“产品经理/运营/数据分析/前端”等少数岗位分支。
- 文档和代码里如果出现类似硬编码倾向，需要主动指出风险，并给出通用化替代方案。这类问题对后续产品扩展很致命。
