# 实时语音对话项目目录框架

## 一、实现方向

当前先创建一个独立原型项目 `realtime-voice-prototype/`，仅实现本地浏览器中的双向实时语音对话：

```text
浏览器麦克风
-> FunASR Paraformer-zh-streaming
-> 流式对话模型
-> 流式 TTS
-> 浏览器播放 AI 语音
```

首版不包含登录、数据库、简历、岗位、反馈报告和录音保存。技术栈建议为：

| 部分 | 建议技术 |
| --- | --- |
| 前端 | `Vite + TypeScript`，负责音频采集、播放、状态与临时转写 |
| 后端编排 | `Python + FastAPI + WebSocket`，便于直接接入 `FunASR` |
| 实时 ASR | `FunASR Paraformer-zh-streaming` |
| 对话模型 | 先抽象为适配器，首轮依据硬件决定本地或国内云模型 |
| 流式 TTS | 先抽象为适配器，候选可验证 `CosyVoice` 或国内云服务 |
| 存储 | V0 不使用数据库，不保存音频或会话记录 |

## 三、目录职责

### `web/`

浏览器交互端，只负责用户可见体验：

- 请求麦克风权限并持续发送音频数据。
- 播放后端返回的 AI 音频流。
- 用户开口时通知后端执行打断。
- 展示连接状态、识别中的文字和 AI 回复文字。

前端不保存密钥，不直接依赖具体模型，也不持久化转写。

### `server/app/session/`

实时对话的控制中心：

- 管理一次会话从开始到结束的状态。
- 判断当前是用户讲话、AI 思考还是 AI 播放阶段。
- 处理用户打断：停止 TTS 播放、取消当前生成、切回听取用户。
- 将转写、回复文本和播放事件推送到网页。

这一层决定产品的“流程和边界可控”，应与模型实现分开。

### `server/app/providers/`

所有模型接入都通过适配器完成：

- `asr/funasr_streaming.py` 负责首期实时识别主链路。
- `llm/` 暂不锁死具体模型，保证之后更换本地或国内云模型无需改会话流程。
- `tts/` 同样保留流式输出接口，以便测试不同语音生成方案的首包延迟。

### `models/`

本地模型文件目录。仅保留下载说明和忽略规则，不将大模型权重提交到项目中。

### `evaluation/`

用于记录真实体验数据，而不是只凭主观感觉判断：

- 开口到首字转写时间。
- 说完到 AI 首段语音播放时间。
- 打断后 AI 停止播报时间。
- 中文、中英混合和噪声环境下的识别表现。

## 四、前后端事件协议

网页与后端通过一个 `WebSocket` 会话通道通信，V0 建议保留以下事件：

| 方向 | 事件 | 用途 |
| --- | --- | --- |
| 前端 -> 后端 | `session.start` | 开始本轮实时对话 |
| 前端 -> 后端 | `audio.chunk` | 连续发送用户麦克风音频 |
| 前端 -> 后端 | `user.interrupt` | 用户打断当前 AI 播报 |
| 前端 -> 后端 | `session.stop` | 结束对话并清除临时上下文 |
| 后端 -> 前端 | `session.ready` | 模型与音频链路就绪 |
| 后端 -> 前端 | `transcript.partial` | 用户临时转写 |
| 后端 -> 前端 | `transcript.final` | 用户本轮确认转写 |
| 后端 -> 前端 | `assistant.text.delta` | AI 流式回复文本 |
| 后端 -> 前端 | `assistant.audio.chunk` | AI 流式音频播放数据 |
| 后端 -> 前端 | `assistant.cancelled` | 当前 AI 回复因打断停止 |
| 后端 -> 前端 | `session.error` | 展示可理解的故障提示 |

## 五、四阶段实施落点

### 阶段 1：实时语音输入与识别

目标：浏览器讲话，页面实时出现 `FunASR` 转写结果。

首批实现目录：

```text
web/src/audio/
web/src/realtime/
web/src/ui/
server/app/api/voice_ws.py
server/app/providers/asr/
server/app/audio/
server/app/protocol/
```

验收：用户可开始/结束麦克风输入，页面看到低延迟的增量转写。

### 阶段 2：流式文字对话

目标：用户说完一轮后，AI 在页面流式输出文字回答。

新增实现目录：

```text
server/app/providers/llm/
server/app/session/
server/app/prompts/
```

验收：用户能够连续多轮说话，AI 回复受基础角色和长度边界控制。

### 阶段 3：AI 流式语音回答

目标：AI 的文字回复被实时转换为声音并播放。

新增实现目录：

```text
server/app/providers/tts/
web/src/audio/audio-player.ts
```

验收：网页中实现完整的“用户语音输入 -> AI 语音回答”闭环，并记录端到端首响延迟。

### 阶段 4：打断与体验优化

目标：用户在 AI 播报中开口，系统立即停止当前回答并进入新一轮输入。

重点实现目录：

```text
server/app/session/turn_controller.py
server/app/session/interruption.py
server/app/audio/vad.py
evaluation/
```

验收：打断稳定生效，转写展示清楚，延迟和错误场景已有可重复的测试记录。

## 六、暂未锁定的两个组件

`FunASR Paraformer-zh-streaming` 已确定为第一阶段识别主链路。为真正开始编码前避免返工，仍需确定：

| 组件 | 待决定问题 |
| --- | --- |
| 对话模型 | 本机是否可运行低延迟本地模型，或首轮调用国内云端流式模型 |
| 流式 TTS | 使用可本地运行的 `CosyVoice`，还是首轮接入国内云端流式 TTS |

这两个选择取决于本机硬件，尤其是显卡型号与显存；但不影响阶段 1 先搭建 `FunASR` 实时转写链路。
