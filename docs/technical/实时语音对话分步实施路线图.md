# 实时语音对话 V0 分步实施路线图

## 一、目标与原则

V0 的目标是在本地电脑浏览器中跑通可体验的实时语音对话：

```text
用户讲话
-> FunASR Paraformer-zh-streaming 实时转写
-> 对话模型流式回复
-> 流式 TTS 输出 AI 语音
-> 用户可打断 AI 并继续交流
```

实施遵循三个原则：

- 每一步只验证一个关键问题，避免多个组件同时出错而无法定位。
- 优先验证影响体验的延迟与打断能力，不提前建设业务功能。
- 每一步都有可演示结果和验收门槛，通过后再进入下一阶段。

## 二、整体阶段划分

| 阶段 | 解决的问题 | 可演示结果 |
| --- | --- | --- |
| 0. 环境与骨架 | 项目能否稳定运行和通信 | 网页连接本地后端并显示状态 |
| 1. 实时转写 | 浏览器语音能否低延迟识别 | 边说边显示用户转写 |
| 2. 文字对话 | 对话模型是否能连续响应且受控 | 说话后 AI 流式输出文字 |
| 3. 语音回答 | AI 回答能否快速播报 | 语音输入后听到 AI 语音 |
| 4. 打断与轮次 | 能否像真实对话一样插话 | AI 播报中可被用户打断 |
| 5. 体验评估 | 方案是否值得进入面试场景 | 形成延迟与可用性结论 |

## 三、阶段 0：环境确认与项目骨架

### 目标

创建可运行的前后端工程，确认浏览器与本地服务可以建立实时连接。

### 要做的事

- 创建 `realtime-voice-prototype/` 工程目录。
- 创建 `web/`：`Vite + TypeScript` 页面。
- 创建 `server/`：`Python + FastAPI + WebSocket` 服务。
- 定义前后端最小事件协议：`session.start`、`session.ready`、`session.stop`、`session.error`。
- 页面实现连接状态区与开始/结束按钮。
- 检查 Python、Node.js、麦克风权限及模型运行环境。

### 交付物

```text
realtime-voice-prototype/
|-- web/
|-- server/
|-- models/
|-- scripts/
`-- evaluation/
```

### 验收标准

- 前端和后端可以分别启动。
- 网页可建立并关闭 `WebSocket` 会话。
- 页面能显示“未连接、已连接、已结束、异常”状态。
- 尚未接入任何模型，也不应出现音频和数据持久化。

### 决策点

本阶段同时确认本机显卡与可用资源。若本地无法支撑后续对话模型或 TTS，不影响 `FunASR` 阶段推进，但应尽早决定这些环节采用国内云端接口。

## 四、阶段 1：浏览器音频输入与 FunASR 实时转写

### 目标

先解决“系统能否实时听懂用户”这一问题，不接 AI 回复和 TTS。

### 要做的事

- 网页申请麦克风权限并读取连续音频流。
- 将浏览器音频转换为 `FunASR` 所需的采样率与格式。
- 通过 `WebSocket` 将音频块传入后端。
- 后端接入 `FunASR Paraformer-zh-streaming`。
- 将增量转写与最终转写返回网页。
- 页面区分“识别中”文本和“已确认”文本。

### 核心文件

```text
web/src/audio/microphone.ts
web/src/audio/pcm-encoder.ts
web/src/realtime/socket-client.ts
web/src/ui/transcript.ts
server/app/api/voice_ws.py
server/app/audio/formats.py
server/app/providers/asr/base.py
server/app/providers/asr/funasr_streaming.py
server/app/protocol/
```

### 验收标准

- 用户说普通话时，网页持续显示增量识别内容。
- 停止说话后，页面能够形成一条最终转写。
- 全程不保存原始音频和历史文本。
- 能记录“开口到首次出现识别文字”的体验延迟。

### 重点测试

| 场景 | 观察内容 |
| --- | --- |
| 安静环境普通话 | 基础识别与首字延迟 |
| 较长连续表达 | 转写是否频繁跳变 |
| 中英文混合 | 产品名、技术名词识别情况 |
| 短暂停顿 | 是否过早结束当前发言 |

### 通过条件

若转写明显跟不上讲话、内容大量错误或终句判断过慢，先优化识别链路，不进入对话模型接入。

## 五、阶段 2：流式文字对话

### 目标

验证“用户说完一句后，AI 能否及时、连续、受控地回复文字”。

### 要做的事

- 定义对话模型统一接口，避免绑定具体厂商或本地模型。
- 用户最终转写确认后，提交给对话模型。
- 将 AI 回复文本以增量方式推送到网页。
- 设置 V0 系统提示词：回答简短、适合口语、允许被打断。
- 建立临时会话上下文，仅保存在内存中。

### 核心文件

```text
server/app/providers/llm/base.py
server/app/providers/llm/selected_provider.py
server/app/session/manager.py
server/app/session/state.py
server/app/prompts/assistant.txt
web/src/ui/transcript.ts
```

### 验收标准

- 用户说一句话后，AI 能在页面流式输出回复文字。
- 支持连续多轮对话，AI 能理解上一轮上下文。
- 点击结束或刷新页面后，会话上下文清空。
- AI 回复受长度和角色边界控制，不出现漫长段落。

### 决策点

对话模型在本阶段必须选定一种可运行实现：

- 本机性能足够时，选择本地流式模型。
- 本机性能不足或延迟不可接受时，选择国内可直连的云端流式模型，接口层保持不变。

## 六、阶段 3：流式 TTS 与 AI 语音回答

### 目标

完成用户说话、AI 用语音回答的完整闭环。

### 要做的事

- 定义 `TTS` 统一流式接口。
- 将 AI 流式文本按适合合成的短句或片段喂给 `TTS`。
- 将返回的音频块发送到网页。
- 网页实现顺序播放、清空队列与音频状态显示。
- 在页面中同时展示 AI 已生成的文字，便于观察播报进度。

### 核心文件

```text
server/app/providers/tts/base.py
server/app/providers/tts/selected_provider.py
web/src/audio/audio-player.ts
server/app/protocol/server_events.py
```

### 验收标准

- 用户发言完成后，可听到 AI 语音回答。
- AI 语音播放与页面文字基本对应。
- 能测量“用户说完到 AI 首段声音播放”的总等待时间。
- 多轮对话中音频不重叠、不乱序、不残留上一轮播放。

### 决策点

首个 TTS 方案应以低首包延迟为优先。若本地 `TTS` 无法达到可接受响应速度，可先换为国内流式 `TTS`，保留未来本地替换能力。

## 七、阶段 4：打断、轮次与对话自然度

### 目标

让用户可以在 AI 说话过程中自然开口，系统及时停止 AI 并处理新输入。

### 要做的事

- 增加 `VAD` 或等效机制，判断用户开始讲话。
- AI 正在播放时检测到用户有效语音，触发 `user.interrupt`。
- 后端取消当前对话生成任务和 TTS 流。
- 前端立即停止当前 AI 播报并清空未播放音频。
- 新的用户输入进入下一轮上下文。
- 区分 AI 已播出文本和因打断未完成的文本。

### 核心文件

```text
server/app/audio/vad.py
server/app/session/turn_controller.py
server/app/session/interruption.py
web/src/audio/audio-player.ts
web/src/realtime/session-state.ts
```

### 验收标准

- AI 说话时，用户开口后 AI 能停止播报。
- 打断后用户的新内容能被正确转写并触发新回复。
- 多次打断不会导致播放队列混乱或会话失效。
- 页面可以看出哪一段 AI 回答被打断。

### 通过条件

若打断频繁误触发或停止明显滞后，需要在进入面试场景前优先解决，因为这会直接破坏实时对话体验。

## 八、阶段 5：体验评估与路线判断

### 目标

用可记录的数据判断 V0 是否达到“值得继续做面试场景”的水平。

### 要做的事

- 编写固定测试对话脚本，覆盖短回答、长回答、主动打断和中英文词汇。
- 至少完成多轮实际体验记录。
- 对 `ASR`、对话模型、`TTS` 和打断分别记录问题归属。
- 汇总结论：继续优化组合链路，还是引入云端端到端方案做体验对照。

### 评估指标

| 指标 | 说明 |
| --- | --- |
| 首字转写延迟 | 用户开始说话到页面出现识别文本的时间 |
| 终句确认等待 | 用户说完到系统确认当前话轮的时间 |
| AI 首字回复延迟 | 终句确认到页面出现 AI 文本的时间 |
| AI 首音频延迟 | 用户说完到听见 AI 回答的时间 |
| 打断停止延迟 | 用户开口到 AI 音频停止的时间 |
| 转写准确体验 | 用户是否认为内容足以支撑对话 |
| 对话流畅体验 | 用户是否愿意持续交流 5 分钟以上 |

### V0 完成标准

- 能够完成至少 5 分钟连续双向语音对话。
- AI 能以语音回答，且用户可有效打断。
- 页面临时转写足以帮助观察对话过程。
- 无录音或转写内容被持久化保存。
- 已形成延迟瓶颈和下一步优化清单。

## 九、实施顺序与不可提前事项

### 推荐顺序

```text
工程骨架
-> FunASR 实时转写
-> 对话模型文字回复
-> 流式 TTS 语音回复
-> 用户打断
-> 延迟评估
-> 求职面试场景
```

### V0 期间不要提前加入

- 登录、注册和数据库。
- 简历与岗位 JD 解析。
- 面试反馈和评分。
- 云端录音存储与历史记录。
- 对外分享和生产部署。

这些功能都不能回答当前最关键的问题：实时语音链路是否足够自然可用。

## 十、进入下一阶段前仍需确认的事项

阶段 0 与阶段 1 可以直接开始，使用 `FunASR Paraformer-zh-streaming` 跑通实时转写。进入阶段 2 和阶段 3 前，需要依据本机性能与实际延迟分别选定：

- 流式对话模型采用本地部署还是国内云服务。
- 流式 `TTS` 采用本地部署还是国内云服务。

无论选择哪种实现，都应遵循已设计的适配器接口，以便后续不改动会话流程即可替换组件。
