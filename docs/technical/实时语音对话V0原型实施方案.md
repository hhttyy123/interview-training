# 实时语音对话 V0 原型实施方案

## 一、原型目标

本阶段只验证一个核心问题：用户是否能在电脑浏览器中，与 AI 进行流畅、低延迟、可打断的实时语音对话。

V0 不承载求职产品的完整业务能力，不接入简历、岗位 JD、面试反馈、账户、付费或历史训练记录。待实时体验验证成立后，再将其接入 AI 求职训练产品。

## 二、已确定的产品边界

| 项目 | V0 决策 |
| --- | --- |
| 使用端 | 电脑浏览器网页 |
| 使用范围 | 本地运行，由本人体验 |
| 核心交互 | 用户与 AI 双向实时语音对话 |
| AI 回应方式 | AI 直接输出语音回答 |
| 打断能力 | 用户可在 AI 说话时开口打断 |
| 文字展示 | 页面临时显示双方实时转写 |
| 数据保存 | 不保存录音、不持久化转写，刷新后清空 |
| 业务功能 | 不做岗位、简历、反馈与付费 |
| 网络要求 | 中国大陆网络环境可直接使用 |

## 三、技术路线决策

### 首版主路线

V0 优先采用可拆分、可本地化的实时语音链路，先搭建三段式核心流程：

```text
浏览器麦克风音频流
-> FunASR Paraformer-zh-streaming 实时转写
-> 对话模型流式生成回复
-> 流式 TTS 合成并播放 AI 语音
```

应用层负责：

- 开始和结束会话。
- 麦克风授权与连接状态显示。
- VAD、轮次状态和用户打断 AI 的交互控制。
- 临时展示双方转写。
- 配置基础对话边界，例如 AI 身份、回复长度与禁止内容。

模型组件负责：

- `FunASR Paraformer-zh-streaming`：持续识别用户语音并输出增量转写。
- 对话模型：接收确认后的用户发言，流式生成 AI 回复文本。
- 流式 `TTS`：尽早将回复文本转换为可播放音频。

### 选型原则

- `ASR` 首选 `FunASR Paraformer-zh-streaming`，优先在本地启动服务。
- 对话模型与流式 `TTS` 通过统一接口接入，首轮可根据本机性能选择本地模型或国内云服务。
- 云端端到端实时语音模型保留为体验对照组，不作为本方案主链路。

选择理由：

- 语音识别、回复生成与语音合成分别可调试、可替换。
- `FunASR` 面向中文流式识别，适合低延迟语音输入验证。
- 更贴近未来国内部署和私有化运行目标。
- 可通过每段链路的指标定位延迟瓶颈。

启动前需确认：

- 本机是否具备可用于本地推理的显卡及显存。
- 对话模型和流式 `TTS` 首轮采用本地还是国内云端接入。
- 本地 `FunASR` 服务运行依赖及模型下载方式。

## 四、V0 功能清单

### 必须实现

- 一个本地网页界面。
- `开始对话` 与 `结束对话` 按钮。
- 浏览器麦克风权限申请。
- 用户语音实时发送。
- AI 实时语音回答与播放。
- 用户在 AI 播报过程中的语音打断。
- 当前连接、录音和 AI 输出状态提示。
- 页面临时展示用户与 AI 的转写文本。

### 本阶段不实现

- 登录注册与用户体系。
- 简历和岗位 JD 上传。
- 面试场景设计与岗位追问。
- 面试评分与反馈报告。
- 音频录制保存与回放。
- 转写历史保存。
- 在线部署和对外分享。
- 面向外部用户的生产级私有化部署。

## 五、最小系统架构

```text
本地浏览器网页
  |-- 麦克风采集
  |-- AI 音频播放
  |-- 打断控制
  |-- 临时转写展示
  |
本地会话编排服务
  |-- WebSocket 音频与事件转发
  |-- VAD / 轮次状态 / 打断取消
  |-- 对话上下文与临时转写
  |
模型服务层
  |-- FunASR Paraformer-zh-streaming：用户语音转写
  |-- 对话模型适配器：流式回复文本
  |-- 流式 TTS 适配器：AI 回复音频
```

原则：模型访问密钥不得直接写入网页前端代码；模型服务通过适配器隔离，以便本地模型与国内云服务互换。

## 六、V0 验收标准

| 验收项 | 判断方式 |
| --- | --- |
| 能够开始会话 | 点击按钮后成功开启麦克风并连接模型 |
| 能够双向说话 | 用户说话后，AI 以语音连续回应 |
| 具备低等待感 | 用户自然交流时无明显影响体验的长停顿 |
| 可以打断 | AI 说话时用户开口，AI 能停止原回答并继续交流 |
| 转写可观察 | 页面可看到双方本次会话的临时文本 |
| 不留本地记录 | 刷新页面后不保留音频和转写数据 |

## 七、后续演进路线

### 路线 A：接入求职面试场景

实时对话体验达标后，再加入：

- 固定岗位与面试官提示词。
- 用户粘贴简历和岗位 JD。
- 围绕岗位的动态追问。
- 面试结束后的结构化反馈。

### 路线 B：国内业务部署

需要提供给测试用户使用时，再加入：

- 国内服务器部署。
- 权限、授权说明与隐私机制。
- 会话数据存储策略。
- 成本与并发监控。

### 路线 C：生产级私有化优化

V0 已按可私有化的拆分链路实现；若未来需要面向真实用户提供服务，则继续优化：

```text
FunASR Paraformer-zh-streaming
-> 私有化或可部署的对话模型
-> 流式 TTS
```

实时语音识别首选 `FunASR Paraformer-zh-streaming`。该模型面向中文流式识别，可持续接收用户语音并输出增量转写，相比依赖音频切片模拟实时效果的 `Whisper`，更适合低延迟中文对话链路。

后续私有化验证应重点评估：

- 用户开始说话后的首字转写延迟。
- 用户结束回答后触发 AI 回复的等待时长。
- 面试术语、项目名称与中英文混合表达的识别准确度。
- 用户打断 AI 时的轮次切换体验。
- 本地运行所需 CPU/GPU 资源及并发成本。

`Whisper` 保留为事后高质量转写或识别效果对照方案，不作为低延迟实时对话的首选 ASR。

## 八、当前结论

V0 不以“搭建完整面试产品”为目标，而以跑通最小实时语音体验为目标：

```text
本地浏览器
-> 双向实时语音
-> AI 语音回答
-> 支持用户打断
-> 页面临时转写
-> 不保存录音和历史内容
```

首版采用 `FunASR Paraformer-zh-streaming + 对话模型 + 流式 TTS` 的可拆分链路，从一开始验证低延迟、可打断和可本地化能力；对话模型与 `TTS` 是否完全本地运行，将依据本机硬件和首轮性能测试决定。
